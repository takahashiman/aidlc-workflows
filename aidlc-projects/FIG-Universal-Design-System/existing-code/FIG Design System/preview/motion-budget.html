Motion Budget

「スピード自体が大きな価値」という FIG の理念を、時間軸の数値契約として可視化する。
semantic.css §17 の --motion-experience-* ロール束を、200ms バジェット境界とともに時間バーで表示。

1. 体感バジェットの時間バー

各ロールの実時間（ms）を 1ms = 2px で可視化。点線は 200ms バジェット境界。例外帯（240ms / 280ms）は警告色で示し、その理由を明記する。

press-result --motion-experience-press-result
150 ms
reveal-quick --motion-experience-reveal-quick
180 ms
navigation --motion-experience-navigation
200 ms
sheet-rise --motion-experience-sheet-rise
240 ms*
modal-rise --motion-experience-modal-rise
280 ms*
dismiss-quick --motion-experience-dismiss-quick
150 ms
dismiss-press --motion-experience-dismiss-press
75 ms
バジェット内（≤ 200ms） 例外帯 — sheet (240ms) 最大例外 — modal (280ms)
例外帯の使用条件：sheet-rise / modal-rise は「高さの登場 + フォーカス遷移」を伴うときだけ許容する。タブ切替やフィルタ適用にこれらを使ってはならない。

2. ライブ実演 — トークンを実際に動かす

各ボタンは対応する --motion-experience-* を transition に注入。下のボックスが動く時間で「体感バジェット」を確認できる。

Move
使い方：ボタンを押すと、対応するロールでボックスが横移動する。
OS で「動きを減らす」を ON にすると、全ロールが 0.01ms に自動上書きされる（@media (prefers-reduced-motion: reduce)）。

3. 規約 — なぜ 200ms なのか

  • Doherty Threshold（IBM, 1982）— 400ms 以内のレスポンスがユーザー生産性を最大化。その半分が 200ms。
  • Material Motion "small" 帯（Google）— 50–200ms を「area-restricted, simple」な動きに割り当てる規約。
  • Apple HIG— "Animations should occur in response to user actions and feel instantaneous"。体感即時の目安として 200ms 前後を提示。
  • FIG ロゴマニュアル「スピード」— ロゴデザイン段階で言語化された理念を、ここで UI コンポーネントに継承する。
詳細規約は patterns/transition-budget.md を参照。CI での違反検出スクリプトは将来課題。
