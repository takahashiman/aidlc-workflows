## 質問（Part 1）

## Question S1 — ストーリーの分解アプローチ（詳しい説明つき・再確認）
#2 のストーリーを「どんな単位・並べ方」で書き起こすかの選び方です。中身（何を作るか）は同じで、**並べ方が変わるだけ**です。

### それぞれの違い（同じ題材を3通りで表すと）
題材＝「BusDelayAlerts の状態色をトークン化する」「ポータルだけで移行操作を完遂する」。

- **A) エピック基盤**: 機能のかたまり（Epic）でグループ化し、その下にストーリーをぶら下げる。
  ```
  Epic D（dogfooding）
    └ US-D3: 開発者として、状態色を semantic トークン化したい（AC: 正常/遅延/運休が token 経由）
  Epic P（ポータル）
    └ US-P2: 閲覧者として、ポータルだけで移行手順を辿りたい（AC: 口頭補足なしで完遂）
  ```
  → **構造が整理され、後の Unit 分解（Construction）にそのまま渡しやすい**。前サイクルと一貫。

- **B) ユーザージャーニー基盤**: 利用者の体験を**時系列の旅**として並べる。
  ```
  旅1「既存アプリを整える」: 閲覧→clone→配置→トークン化→状態色→signature→ビルド確認→終了通知
  旅2「ポータルだけで操作する」: 入口→役割選択→ガイド→移行→バージョン参照→昇格提案
  ```
  → **流れ・導線が直感的**（ポータル綿密化＝画像02 の2シナリオと相性◎）。ただし Unit 分解時に再整理が要る。

- **C) ハイブリッド**: **A で構造化しつつ、B の旅を「通し線」として併記**。
  ```
  Epic D/P/X でストーリーを定義（=A）
  ＋ 付録として「旅1/旅2」のステップ列に各ストーリーを並べ直したマップ（=B）
  ```
  → 構造（Unit 分解向き）と 導線（ポータル説明向き）の**両取り**。作業はやや増える。

### おすすめの考え方
- 今回は **②ポータル綿密化＝"ポータルだけで操作完結（＝旅）"** が中核で、画像02/03 も**フロー図**です。
  一方で Construction の **Unit 分解には Epic 構造が便利**。よって **C（ハイブリッド）** が両目標に最も噛み合います。
  （前サイクルとの一貫性だけを最優先するなら A。）
