# Personas — FIG デザインシステム循環システム

> 5ペルソナ（SQ1: Core Engineer と 拡張プロジェクト開発者 を「エンジニア」に統合）。

## P1. Core Maintainer（デザイナー）
- **アーキタイプ**: デザインシステムの番人。FIG ブランドの一貫性と普遍性を守る最終責任者
- **目的(Goals)**: Core DS の品質・一貫性維持／適切な Core 昇格判断／taxonomy の整理・承認
- **課題(Pain points)**: レビュー負荷の集中、普遍化作業の手間、ブランド逸脱の検知漏れ
- **技術文脈**: デザインツール、GitHub Issue/PR レビュー、限定的なコード読解（トークン/CSSは読める）
- **主な関与**: Core昇格レビュー、taxonomy ガバナンス、ドッグフーディングの普遍化判断

## P2. エンジニア（Core 実装＋拡張開発を兼務）
- **アーキタイプ**: フロントエンド中心のエンジニア。Core DS も各製品の拡張も実装する
- **目的**: 高速な UI 構築、トークン経由の正しい実装、Core を pin した再現性確保
- **課題**: 三層アーキ違反の混入、バージョン不整合、無関係コードによるノイズ
- **技術文脈**: React/JSX、CSS カスタムプロパティ、git submodule、CI/CD、各チーム標準の AI アシスタント
- **主な関与**: Core/拡張の実装、CI/CD（Lint/VRT/バージョン収集）、template 派生

## P3. 既存プロジェクト移行担当
- **アーキタイプ**: 既存製品を Core DS へ寄せるリード/エンジニア
- **目的**: 段階的に移行し、画面内混在を避けつつ、完了基準を満たす
- **課題**: レガシー画面の負債、新旧混在リスク、完了判定の曖昧さ
- **技術文脈**: 既存コードベース、移行チェックリスト、Core DS submodule
- **主な関与**: 既存取り込み、段階移行、完了判定

## P4. AI エージェント（各チーム標準: Cursor / Claude 等）
- **アーキタイプ**: 開発を代行する非人間アクター
- **目的**: 使い方ページ＋プロンプトに従い、正確に複製/生成し、三層アーキを遵守する
- **課題**: コンテキスト過多・無関係情報、ハルシネーション、規約逸脱
- **技術文脈**: スコープ分離されたリポジトリ、常時コンテキスト規約、Interactive Prompt Generator
- **主な関与**: template 複製・初期設定の自律実行、コンポーネント生成、ショーケース収集

## P5. ポータル閲覧者（社内デザイナー / PdM / 新規参加者）
- **アーキタイプ**: 資産を探す・学ぶ・参照する利用者（玄人〜新規参加者まで）
- **目的**: 目的プロジェクト/コンポーネント/デモへ即時到達、使い方理解、参照バージョン把握
- **課題**: 情報過多で目的物に辿り着けない、最新かどうかの不安
- **技術文脈**: ブラウザ（PC / Web-Admin）、GitHub Pages
- **主な関与**: ポータル閲覧、使い方参照、ショーケースからの発見
